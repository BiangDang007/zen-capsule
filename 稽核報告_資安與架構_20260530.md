> 產出日期：2026-05-30

## 0. 一句話總結

這是一個完成度相當高、且**已經做了大量正確資安強化**（XML prompt 包裝、Serializable 交易防 race、CSV 注入防禦、DB-backed 登入鎖定、帳號刪除符合 GDPR）的專案。但在**上架前**仍有 3 個 Critical 與數個 High 等級問題必須處理，最致命的是：**release keystore 密碼被 commit、付費機制形同虛設、免費用戶可繞過用量限制、refresh token 可當 access token 用**。Google Play 對「攔截所有 App 通知並送往第三方 LLM」的政策審查是最大的上架不確定因素。

## 2. CRITICAL（上架前必須修）

### C1. Release 簽章密碼被 commit 進 git
- **問題**：Android 上架簽章的 store 密碼與 key 密碼以明文寫在被 git 追蹤的檔案，且兩者同密碼、弱且可猜。
- **證據**：`packages/mobile/android/gradle.properties:47-50`
  ```
  ZEN_RELEASE_STORE_PASSWORD=zencapsule2026
  ZEN_RELEASE_KEY_PASSWORD=zencapsule2026
  ```
  `git ls-files | grep gradle.properties` → 已追蹤。諷刺的是 `app/build.gradle:97-101` 的註解明明叫你放到 repo 外的 `~/.gradle/gradle.properties`。
- **補充（好消息）**：`release.keystore` 二進位檔本身有被 `.gitignore`（`*.keystore`）擋住，未進 repo。所以外洩的是「密碼」不是「金鑰檔」——但密碼弱又同碼，一旦金鑰檔在別處外流即可被冒簽。
- **影響**：若金鑰檔外洩，攻擊者可簽出「看似官方」的惡意更新。
- **修法**：刪掉這 4 行 → 移到 `~/.gradle/gradle.properties` 或 CI secret → **更換兩組密碼** → 若曾 commit 進歷史需 `git rm --cached` 並清歷史。

### C2. 付費機制形同虛設（沒有任何路徑會把使用者升級成 PRO）
- **問題**：全 repo 沒有任何程式會設定 `user.plan = PRO`，也沒有 Google Play Billing 收據驗證、webhook 或後台授權端點。`billing.ts` 只會「讀」狀態並對外宣傳 $4.99 價格。
- **證據**：`packages/backend/src/routes/billing.ts`（整檔只有 `GET /billing/status`）；`usage.service.ts:30-36,70-79` 只讀 `plan`。價格寫在 `billing.ts:30-32`。
- **影響**：要嘛沒人能真的付費升級（功能斷裂），要嘛只能手動改 DB（無稽核、無收據）。若日後補一個「客戶端說我是 PRO」的端點而不做伺服器端收據驗證，使用者就能自助升級。
- **修法**：實作 Google Play Billing 並在**伺服器端**用 `purchaseToken` 對 Play Developer API 驗證後才寫入 `plan=PRO` + `planExpiresAt`；永不信任客戶端自稱的方案。

### C3. 免費用戶可用 `keywordScore` 完全繞過用量限制，並自封「緊急穿透」
- **問題**：`/ai/analyse` 只要客戶端帶了 `keywordScore` 就**直接 return**，完全不經 `checkAndIncrement()`；而且 `score / category / shouldBreakthrough` 全採信客戶端輸入。
- **證據**：`packages/backend/src/routes/ai.ts:142-187`（`if (body.data.keywordScore != null)` 區塊在 `:190` 的 `checkAndIncrement` 之前就 return）；`keywordScore` 是客戶端可控欄位（`analyseSchema`, `ai.ts:18`）；Android 端自己塞值（`ZenNotificationListener.kt:296`）。
- **影響**：(a) 每個請求都帶 `keywordScore` → 永遠不耗用量配額，付費模型的經濟基礎崩潰；(b) 自塞 `score:100` → `shouldBreakthrough` 為真，繞過「專注防火牆」本身。
- **修法**：keyword / log-only 請求也要計入（較便宜的）配額或做伺服器端速率限制；不要讓客戶端決定 `shouldBreakthrough`/`category`，改由伺服器推導或夾限。

## 3. HIGH

### H1.（子 session 找到，已複核）Refresh token 可當 access token 使用
- **問題**：認證守門只呼叫 `req.jwtVerify()`，從不檢查 `type`。refresh token 與 access token 同密鑰簽署且都帶 `sub`，因此 30 天的 refresh token 可當 bearer access token 打所有受保護路由。
- **證據**：`middleware/auth.ts:11-17`（只有 `jwtVerify`，無 claim 檢查）；access token `{ sub }`（`auth.ts:205-208`）、refresh token `{ sub, type:'refresh' }`（`auth.ts:209-211`）。
- **影響**：長效 refresh token 等同 30 天 API 憑證；`/auth/refresh` 也只靠 DB 字串查找（`auth.ts:118`），未驗章。
- **修法**：access token 加 `type:'access'` claim，`authenticate` 拒絕非 access token；`/auth/refresh` 驗證 `type`。

### H2. EXPORT_KEY 為單一全域金鑰
- **證據**：`ai.ts:49-56` `verifyExportKey` 比對 `process.env.EXPORT_KEY`，用於 `:294,:381`；範例值 `.env.example:17` = `change-me-in-production`。
- **澄清（好消息）**：匯出**有**以呼叫者自己的 `userId` 隔離（`ai.ts:301-304,385-386`），且用 `X-Export-Key` header 而非 query、含 CSV 注入防禦——所以**不是**跨用戶外洩，而是「弱憑證 / 無輪替 / 無 constant-time 比較」問題。
- **修法**：改用使用者上的 admin 角色旗標，或至少用高熵可輪替金鑰 + `crypto.timingSafeEqual` + 稽核紀錄。

### H3. Google Play 通知監聽 / 資料安全揭露風險（最大上架不確定因素）
- **問題**：服務攔截**所有 App** 的通知標題＋內文，送到後端再轉送 Anthropic。Play 對 `BIND_NOTIFICATION_LISTENER_SERVICE` 有嚴格限制，且禁止在非核心功能下傳輸通知內容。
- **證據**：`ZenNotificationListener.kt:19-28,113-188`（攔截全部、`:158` 取消原通知、`:278/:315` 送網路）；`AndroidManifest.xml:31-39`。
- **隱私政策對齊（好消息）**：`docs/privacy-policy.html:36,43,58` 有揭露送往 Anthropic、不用於訓練、傳輸加密。
- **缺口**：
  1. 政策說內容「temporarily sent（暫時傳送）」，但後端把完整 `subject/preview` **長期存進 `BehaviorLog`**（`ai.ts:236-259`，理由是「改善 AI 準確度」）——與「暫時」矛盾。
  2. Play 要求此功能必須是 App 的**核心目的**，且可能需要通知存取宣告表單 / 人工審查；把他人訊息內容送往第三方 LLM 屬高度審查樣態。
  3. Data Safety 表單必須申報蒐集「訊息 / 其他 App 內內容」並與第三方分享。
- **修法**：修正「暫時」措辭、加上保存期限與刪除機制、確認 Data Safety 表單申報、準備通知存取審查。

### H4. CSP 含 `'unsafe-inline'` + 三個 production host 不一致
- **CSP**：`index.ts:88` `style-src 'self' 'unsafe-inline'`（影響不大，僅靜態頁）。
- **Host 漂移（真正的維運風險）**：
  - CORS 白名單：`https://app.zencapsule.com`（`index.ts:60`）
  - Android release base：`https://zen-capsule-fullstack-production.up.railway.app`（`ZenNotificationListener.kt:38`、`api.ts:5`）
  - README 寫 release 是 `https://api.zencapsule.com`（`README.md:77`）
- **影響**：三個不同的正式網域，CORS 可能與實際客戶端對不上；文件與實作不符。
- **修法**：統一一個正式網域，對齊 CORS / 客戶端 / README；移除 `'unsafe-inline'` 或改用 hash。

## 4. MEDIUM

### M1. Per-IP 登入鎖定可被繞過，也可被濫用反鎖他人
- **證據**：`auth.ts:19-36,75-96` 僅以 `req.ip` 計數，無 per-account 節流；rate-limit `keyGenerator`（`index.ts:78`）在登入前 `req.user.sub` 為 undefined，等同也只看 IP。
- **影響**：(a) 攻擊者輪換 IP 即可暴力破解任一帳號（沒有 per-account 限制）；(b) 在反向代理後若未設 `trustProxy`，所有人共用代理 IP → 一人可鎖全站；若 `trustProxy` 設錯，攻擊者可偽造 `X-Forwarded-For` 規避。
- **修法**：加 per-account 失敗計數 + 指數退避；確認 Railway 上 `trustProxy` 與部署拓撲相符；N 次失敗後加 captcha。

### M2. Prompt injection 防護偏淺
- **證據**：`urgency.service.ts:46-52` 的 `sanitise()` 只把 `< >` 換成全形並截長度；未中和標籤「文字」、換行或指令式內容。
- **緩解（好消息）**：system prompt 明示把標籤內容當資料（`:21-23,:152-155`）；JSON 解析失敗時 fail-safe 封鎖（`:103-114,:196-201`）；伺服器端再夾限 `shouldBreakthrough`（`:133-140`）。
- **影響**：最壞情況是攻擊者讓自己的通知衝高分穿透專注（騷擾 / 社交工程），非資料外洩（模型無工具、無祕密）。
- **修法**：也過濾標籤名稱字面（如 `user_content`）、限制換行、並用 Zod 驗證解析後物件（夾 `score` 0–100、`category` enum）。

### M3.（子 session 找到，已複核）`.env.example` 內含預設不安全值，且防呆常數寫錯
- **證據**：`.env.example:5` `JWT_SECRET=change-me-in-production`、`:17` `EXPORT_KEY=change-me-in-production`；但 `index.ts:28` 的防呆只擋字面 `'dev-secret-change-me'`——**和範例值不符**，等於形同虛設。
- **影響**：正式環境可帶著已知佔位 `JWT_SECRET` 開機（可偽造 token）而防呆抓不到。
- **修法**：拒絕所有已知佔位值，並在 production 強制 `JWT_SECRET` 最小長度 / 熵。

### M4. PORT 文件與實作不一致（3000 vs 3001）
- **證據**：`.env.example:13` 與 `index.ts:138` 預設 `3000`；但 README / CLAUDE.md 與客戶端（`ZenNotificationListener.kt:37`）用 `3001`。
- **修法**：`.env.example` 與 `index.ts` 預設統一改 3001。

### M5. 後端零測試；shared 需手動 rebuild 無任何強制
- **證據**：只有 `packages/mobile` 有 `jest`，全 repo 找不到 `*.test.ts`；`CLAUDE.md` 要求改 `shared` 後手動 `npm -w packages/shared run build`，backend/mobile 都 import 其 `dist/`。
- **影響**：用量限制、urgency 夾限、auth、CSV escape 這些資安關鍵邏輯沒有測試；`shared/dist` 可能殘留舊版本悄悄上線。
- **修法**：補後端單元 / 整合測試（尤其 C3、H1、M2）；CI 加一步重建 shared 並在 `dist` 不乾淨時 fail。

## 6. 開發流程（你問的「開發步驟哪裡有問題」）

1. **祕密管理沒有從第一天就分層**：keystore 密碼進了 repo、`.env.example` 用真的會被誤用的佔位字串、`.claude/settings.local.json` 內留有大量歷史指令（雖非祕密但暴露其他專案路徑與 curl 範例）。→ 建議：祕密一律走環境變數 / CI secret，repo 內只放 `.example`，並做一次 secret scan（如 gitleaks）。
2. **文件與實作各走各的**：port（3000/3001）、production host（三個）、README 寫 `api.zencapsule.com` 但程式用 Railway 網址。CLAUDE.md 的「Golden Rule」要求改環境就更新 README，但實際已漂移。→ 單一事實來源（一個 config 檔），文件自動對齊。
3. **「先求有」的端點留下半成品**：billing 只有讀、sync 還是 polling（WebSocket 只在註解）、付費升級沒人寫。→ 上架前要嘛把 PRO 拔掉只留 FREE，要嘛把 Play Billing 串完，不能留「假付費牆」。
4. **沒有測試與 CI 守門**：資安強化做了很多，但沒有任何自動化測試去防止它們被改壞；shared 靠人工 rebuild。→ 加最小 CI：build + 重建 shared + 幾個關鍵路徑測試。
5. **平台政策應在開發早期介入**：通知監聽是整個 App 的核心，但 Play 對它的審查最嚴。應在開發初期就確認可過審、把保存期限 / Data Safety / 隱私政策一次對齊，而不是上架前才補。
